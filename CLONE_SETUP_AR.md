# دليل استنساخ سيمبل (SimpleAI) — نسخة كاملة للجهاز الثاني

هذا الملف يشرح كل شي تحتاجه عشان تشغّل نسخة كاملة من الموقع على جهازك الثاني — سواء تبي تكمّل التطوير على نفس النظام الحيّ، أو تبني نسخة مستقلة تماماً.

---

## 1) وش داخل هذي النسخة؟

كل ملفات الموقع موجودة (الواجهة + الوظائف الخادمية + الأصول + ملفات براءة الاختراع والدليل):

- **صفحات الموقع (HTML):** `index.html`, `login.html`, `signup.html`, `brand.html`, `company.html`, `campaigns.html`, `creator.html`, `creators.html`, `admin.html`, `myteam.html`, `dues.html`, `profile.html`, `workflow.html`, `analytics.html`, `top.html`, `campaign.html`, `cart.html`, `members.html`, `team.html`, `terms.html`, `privacy.html`, `share.html`, `c.html` ...
- **منطق مشترك (JS):** `supabase-config.js` (الاتصال بقاعدة البيانات + دوال البيانات), `translations.js`, `i18n.js`, `notifications.js`, `push-subscribe.js`, `splash.js`, `sw.js` (Service Worker), `push-sw.js`, `install-guide.js`.
- **الوظائف الخادمية (Vercel Serverless) داخل مجلد `api/`:**
  - `negotiate.js` — وكيل التفاوض بالذكاء الاصطناعي (Claude).
  - `generate-brief.js` — توليد البريف الإعلاني بالذكاء الاصطناعي.
  - `team-member.js` — إنشاء أعضاء الفريق (Admin API).
  - `start-campaign.js` — بدء الحملة.
- **إعداد النشر:** `vercel.json` (سياسات الكاش)، `manifest.json` (PWA)، الأيقونات.
- **مجلدات إضافية (غير منشورة على الموقع):** `patent/` (براءة الاختراع)، `guide/` (دليل الواتساب)، `partnership/` (ملف شراكة تيك توك). ملاحظة: هذي الثلاثة **مستثناة من GitHub** عبر `.gitignore`، فهي موجودة في هذي النسخة المضغوطة فقط.
- **ملفات SQL جزئية:** `campaign_capacity.sql`, `snapshot.sql`, `update_my_profile.sql` (ليست كامل قاعدة البيانات — انظر القسم 4).

**مهم:** الموقع «ثابت» (static) — ما فيه خطوة بناء (build) ولا `node_modules`. أي خادم ملفات ثابت يشغّله.

---

## 2) تشغيل سريع على الجهاز الثاني (للتطوير/المعاينة)

الكود يحتوي أصلاً على رابط قاعدة البيانات ومفتاح anon العام داخل `supabase-config.js`، فبمجرد ما تشغّل الملفات **يتصل تلقائياً بنفس النظام الحيّ ويعرض نفس البيانات**. ما تحتاج تسوي شي للباك إند عشان يشتغل.

الطريقة الأسهل — من داخل مجلد المشروع:

```bash
# بايثون (موجود بأغلب الأجهزة)
python3 -m http.server 8000
# ثم افتح: http://localhost:8000

# أو عبر Node
npx serve .
```

> الوظائف داخل `api/` (التفاوض، البريف، إضافة عضو) **ما تشتغل** مع خادم ثابت بسيط — تحتاج Vercel (القسم 3) أو `vercel dev`.

---

## 3) النشر الكامل على Vercel (نسخة تعمل بالكامل مثل الأصل)

1. ارفع الكود على مستودع GitHub جديد (أو استنسخ المستودع الحالي).
2. في [vercel.com](https://vercel.com): New Project → اربط المستودع → Deploy (بدون إعدادات بناء — مشروع static + api).
3. أضِف **متغيّرات البيئة** (Settings → Environment Variables). المطلوب اثنان فقط:

   | المتغيّر | الاستخدام | من وين تجيبه |
   |---|---|---|
   | `ANTHROPIC_API_KEY` | وكيل التفاوض + توليد البريف | لوحة Anthropic (console.anthropic.com) |
   | `SUPABASE_SERVICE_ROLE_KEY` | العمليات الخادمية (إقفال الصفقات، إنشاء الأعضاء…) | Supabase → Project Settings → API → service_role |

4. (اختياري) اربط دومينك الخاص من Vercel → Domains.

> رابط Supabase ومفتاح anon مضمّنان في `supabase-config.js` (عامّان وآمنان للمتصفح). المفتاح السرّي (service_role) **لا يوضع في الكود إطلاقاً** — فقط في متغيّرات Vercel.

---

## 4) الباك إند (قاعدة البيانات Supabase) — أهم نقطة

قاعدة البيانات (الجداول، سياسات RLS، الدوال، الـtriggers، صناديق التخزين، وكل البيانات) **تعيش في مشروع Supabase السحابي، وليست داخل هذا الكود.** أمامك مساران:

### المسار (أ): نفس الباك إند (الأسهل)
لا تسوي شي. النسخة على الجهاز الثاني تتصل بنفس مشروع Supabase الحالي وتشتغل على نفس المستخدمين والحملات مباشرة. مناسب لو تبي «تكمّل تطوير» أو «تدير نفس الموقع» من جهاز ثاني.

### المسار (ب): نسخة مستقلة تماماً (قاعدة بيانات منفصلة)
لو تبي نظاماً منفصلاً بالكامل (بيانات مستقلة، لا تأثير على الحيّ):
1. أنشئ مشروع Supabase جديد.
2. طبّق **مخطط قاعدة البيانات الكامل** — جاهز في الملف **`db/schema.sql`** داخل هذي النسخة.
   افتح Supabase → SQL Editor → الصق محتوى الملف كاملاً → Run (مرة واحدة على مشروع فاضٍ).
   يُنشئ لك: الجداول (٢٢) + القيود + الفهارس + الـViews + الدوال (٤٣) + الـTriggers +
   تفعيل RLS + كل السياسات (٤٨) + صناديق التخزين وسياساتها. (بنية فقط، بدون بيانات.)
3. صناديق التخزين (`avatars` عام، `cr-docs` خاص، `content` عام) تُنشأ تلقائياً ضمن الملف.
4. بدّل `SUPABASE_URL` و`SUPABASE_ANON_KEY` في `supabase-config.js` بقيم المشروع الجديد.
5. حدّث متغيّرات Vercel بمفتاح service_role الجديد + `ANTHROPIC_API_KEY`.
6. تعديلات يدوية بعد التنفيذ (موضّحة داخل `db/schema.sql`):
   - الدالتان `simbl_run_reminders` و`simbl_push_on_notification`: بدّل رابط المشروع ومفتاح anon
     (تخص إشعارات push عبر Edge Function اسمه `send-push`).
   - إيميل الأدمن `hello@agentsimpleai.com` مذكور في الدوال والسياسات — بدّله بإيميل الأدمن الجديد.
   - الدالتان `team_overview` و`team_brand_detail` فيهما إيميلات علامات محدّدة (reef/osma/buyar) — عدّلها أو احذفها.
7. (اختياري) لو تبي نسخة من **البيانات** نفسها أيضاً (مع أنها تحتوي بيانات مستخدمين شخصية) — أقدر أصدّرها لك بشكل منفصل.

---

## 5) صناديق التخزين (Storage Buckets)
- `avatars` — صور حسابات المعلنين (عام).
- `cr-docs` — مستندات السجل التجاري للشركات (خاص).

عند إنشاء مشروع منفصل، أنشئ نفس الصناديق بنفس الأسماء.

---

## 6) ملاحظات أمان
- لا تنشر مفتاح `service_role` في الكود أو في مستودع عام — فقط في متغيّرات البيئة.
- مفتاح anon عام بطبيعته ومحميّ بسياسات RLS على الجداول.
- خاصية «حماية كلمات السر المسرّبة» مفعّلة في Supabase — فعّلها في أي مشروع جديد.
